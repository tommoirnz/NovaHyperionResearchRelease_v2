🐛 Bugs & Issues Found

1. 🔴 {location} Placeholder Never Substituted (High Priority)

Your location is {location}. Search for current weather conditions there.

This looks like an f-string template that never gets formatted. If SYSTEM_PROMPT is a class-level string constant rather than an f-string, {location} stays as a literal — meaning I'd always search for a place called {location}. Deeply ironic for a location-aware assistant.

Fix:

SYSTEM_PROMPT = SYSTEM_PROMPT_TEMPLATE.format(location=self.suburb or "Birkdale")

Or make it an f-string at instantiation time, not at class definition time.

2. 🔴 \u2212 Minus Sign Missing from Unicode Replacements (High Priority)

replacements = {
    "\u201C": '"', "\u201D": '"',
    "\u2018": "'", "\u2019": "'",
    ...
}

The Unicode minus sign (−, \u2212) is visually identical to a hyphen but is not a valid Python operator. If it appears in generated code, arithmetic will silently fail or raise a SyntaxError. Also missing:

  • \u00B0 → ° (degree symbol — relevant for weather/math)
  • \u03B1–\u03C9 → Greek letters (relevant for equations)

Fix: Add to the replacements dict:

"\u2212": "-",   # minus sign → hyphen-minus
"\u00B0": " degrees",

3. 🟠 Bare except in Network Check (Medium Priority)

try:
    r = requests.get(url, timeout=2)
    if r.status_code == 200: return True
except:
    continue

Catches everything — including KeyboardInterrupt, SystemExit, and memory errors. Classic Python anti-pattern.

Fix:

except (requests.RequestException, OSError):
    continue

4. 🟠 search_handler=None Passed to CodeExecutionLoop (Medium Priority)

self.smart_loop = CodeExecutionLoop(
    ai_model=self.ai, sandbox=self.code_window,
    search_handler=None, ...)

If CodeExecutionLoop ever calls search_handler(...) without a null check, this throws TypeError: 'NoneType' is not callable. Needs either a guard inside CodeExecutionLoop or a proper handler passed at construction.

Fix (inside CodeExecutionLoop):

if self.search_handler is not None:
    self.search_handler(query)

5. 🟡 Redundant topmost Attribute Set Twice (Low Priority)

self.root.attributes("-topmost", False)
self.root.after(10, lambda: self.root.attributes("-topmost", False))

Setting -topmost to False twice — once immediately, once after 10ms — suggests this was originally True and was patched out hastily. The after() call is now dead weight.

6. 🟡 withdraw() Called on a Possible Frame (Low Priority)

self._cw_frame = tk.Frame(self.root, bg="#0C1219")
self.code_window = CodeWindow(self._cw_frame, ...)
...
self.code_window.withdraw()

tk.Frame doesn't have a withdraw() method — that belongs to Toplevel/Tk. If CodeWindow is a Frame subclass, this call will silently fail or raise AttributeError. Worth verifying whether CodeWindow wraps a Toplevel internally.
